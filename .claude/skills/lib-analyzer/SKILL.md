# 软件流行度评估专家

评估开源库/框架的流行程度、维护状态和生态地位，辅助技术选型决策。

## 触发词
- "评估流行度"、"分析库"、"lib popularity"
- "库健康检查"、"依赖安全吗"、"能用于生产吗"
- "Python 库分析"、"Node.js 库评估"、"Go 包分析"

---

## 通用评估维度

### 1. 基础数据
| 数据项 | 来源 |
|-------|------|
| GitHub Stars、Forks | github.com/{owner}/{repo} |
| 最近提交时间 | GitHub Commits 页面 |
| 开放 Issues 数量 | GitHub Issues 页面 |
| Contributors 数量 | GitHub Insights |
| License 类型 | GitHub repo 头部 |

### 2. 下载趋势
| 语言 | 数据来源 |
|-----|---------|
| Python (PyPI) | pypistats.org、pypi.org/pypi/{pkg}/stats |
| Node.js (npm) | npmjs.com/package/{pkg}、npmtrends.com |
| Go (pkg.go.dev) | pkg.go.dev、goproxy.cn |
| Rust (crates.io) | crates.io/crates/{crate} |

### 3. 维护状态
- **维护者活跃度**：近期提交频率、PR 响应时间
- **版本发布节奏**：最近一次 release 距离现在的时间
- **背书方**：是否被大公司/基金会支持（如 Python Software Foundation、Google、Microsoft）
- **安全响应**：是否有安全政策、已知漏洞修复速度

### 4. 生态地位
- 是否被主流框架/大项目依赖（查看 GitHub Dependents）
- Stack Overflow 相关问题数量和质量
- 是否有替代方案、是否是事实标准

**计算维度与权重**：

**评分等级**：
- 🌟 90-100: 现象级（行业标准，如 React、Express、requests）

- 🟢 50-69: 广泛使用（稳定可靠，广泛采用）
- 🟡 30-49: 有一定采用度（适合特定场景）
- 🔵 <30: 小众或新兴（需要谨慎评估）

## 重要原则

1. **数据优先**：始终引用可查证的数据源，不做无依据猜测
2. **趋势判断**：结合短期（30天/周）和长期（1年）数据判断趋势
3. **语境理解**：考虑该库所在领域的特点（如新兴领域库Stars普遍较低）
4. **保守评估**：数据不足或信息过时时，建议"谨慎评估"
5. **跨源验证**：关键数据（如下载量）尽量从多个来源交叉验证
6. **流行度评分**：优先使用量化评分作为客观参考，同时说明评分依据和局限性
